DOCS · REFERENCE

The Arcana reference

Everything you need to install the terminal, sign in with device pairing, and route model requests through the proxy. The CLI is the operator; the proxy is the bearer.

Last reviewed 2026-07-20 · Source on GitHub · v0.3.5

Install

Install the CLI with your preferred package runner. The published package is arcana-ai; the binary is always arcana.

# Quick start (downloads the binary on first run)
npx arcana-ai

# Global install
npm install -g arcana-ai
# or: pnpm add -g arcana-ai
# or: bun add -g arcana-ai

# From source (dev)
git clone https://github.com/Lento47/arcana && cd arcana
bun install
bun link   # from packages/arcana/

Confirm the install:

arcana --version
# > 0.3.5

The CLI stores state under ~/.arcana/ (or ARCANA_HOME):

  • ~/.arcana/ account store — console login tokens from device flow.
  • Provider keys — usually environment variables (OPENAI_API_KEY, ANTHROPIC_API_KEY, …) or arcana providers.
  • Session history and local DB under the Arcana data directory.
  • Workspace trust records at ~/.arcana/workspace-trust.json after arcana trust.

First login

Pair a terminal with your Arcana account using OAuth RFC 8628 device flow. The CLI shows a one-time seal code, opens your browser, and waits for approval.

arcana console login
# default server: https://arcana.otnelhq.com
# override: arcana console login https://arcana.otnelhq.com

# Terminal shows a seal (user_code), e.g. 3D8C-CVQA
# Browser opens: https://arcana.otnelhq.com/auth/device?code=3D8C-CVQA
# awaiting the seal…
  1. Open the gate URL if the browser did not open automatically.
  2. Sign in (email + password, GitHub, or Google).
  3. The page confirms pairing. The CLI stores the account and license bearer.
  4. Close the browser tab when the terminal shows bound / welcome.

Related console commands:

arcana console orgs      # list orgs for logged-in accounts
arcana console switch    # switch active org
arcana console logout    # remove a stored account
arcana console open      # open the active console in a browser
Pairing is per-machine

The license key is a bearer token. Anyone with read access to your Arcana data directory can act as you. Treat it like an SSH private key — never commit, never share.

Add a key

Provider keys let Arcana call OpenAI, Anthropic, Google, and other vendors. The usual path is environment variables; the CLI also exposes provider management.

# Environment (recommended for CI / shells)
export OPENAI_API_KEY=sk-...
export ANTHROPIC_API_KEY=sk-ant-...
export GEMINI_API_KEY=...

# Interactive / stored credentials
arcana providers

# License / machine activation (separate from console login)
arcana license status
arcana license activate <key>

After console login, the proxy license is used for routed models. Direct provider keys still work for BYOK paths when configured.

arcana run

Start an interactive TUI session (or one-shot run, depending on flags). Loads your agents, tools, skills, and session memory for the current project.

arcana run
arcana run "summarize the auth flow"
arcana run --model <provider/model>
arcana models          # list models first

Common companion commands: arcana session, arcana export, arcana stats, arcana trust (project plugins/tools).

arcana console login

Pair the terminal with your Arcana account via device flow. See First login.

arcana console login [server]   # default: https://arcana.otnelhq.com
arcana console orgs
arcana console switch
arcana console logout [email]
arcana console open

arcana models

List configured providers and models available to this install.

arcana models

arcana providers

Manage provider credentials and connected backends (OpenAI, Anthropic, etc.). Prefer env vars for headless setups.

arcana providers
export OPENAI_API_KEY=sk-...
export ANTHROPIC_API_KEY=sk-ant-...

arcana session

List and work with past sessions (export, continue, inspect).

arcana session list
arcana export
arcana import

arcana stats

Usage statistics for local activity and proxy-backed runs where available.

arcana stats

arcana serve

Start a headless Arcana HTTP server (local control plane). Defaults to loopback; non-loopback binds require ARCANA_SERVER_PASSWORD.

arcana serve
arcana serve --hostname 127.0.0.1 --port 4096

For OpenAI-compatible proxy traffic, use the hosted proxy at https://proxy.arcana.otnelhq.com with your license bearer (see below).

arcana trust

Trust this workspace so project plugins, custom tools, agents, and local MCP can load (security gate).

arcana trust                 # trust cwd
arcana trust status
arcana trust list
arcana trust revoke
# Dev escape: ARCANA_DISABLE_WORKSPACE_TRUST=1

Proxy API · Authentication

Base URL: https://proxy.arcana.otnelhq.com. Authenticated routes expect:

Authorization: Bearer <license_key>

License keys look like ARCANA-PRO-… and are issued by device-flow login (or admin mint). Supabase access tokens and admin secrets are also accepted on some paths; end users should use the license key from console login.

curl -H "Authorization: Bearer $ARCANA_LICENSE" \
     https://proxy.arcana.otnelhq.com/v1/balance
Do not send provider API keys as the proxy bearer.

OpenAI/Anthropic keys belong in your local environment or arcana providers. The proxy bearer is your Arcana license (or JWT for workspace UI), not sk-….

POST /v1/chat/completions

OpenAI Chat Completions-compatible. Streams SSE by default. Accepts the full OpenAI parameter set: model, messages, temperature, top_p, stream, tools, tool_choice, response_format, seed, etc.

POST /v1/chat/completions License or provider key

Example

curl -N -X POST https://proxy.arcana.otnelhq.com/v1/chat/completions \
  -H "Authorization: Bearer $ARCANA_LICENSE" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "anthropic/claude-sonnet-4",
    "messages": [
      { "role": "user", "content": "summarize RFC 8628 in one paragraph" }
    ],
    "stream": true,
    "temperature": 0.3
  }'

Response (non-streaming)

{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "created": 1721000000,
  "model": "claude-sonnet-4-5",
  "choices": [
    {
      "index": 0,
      "message": { "role": "assistant", "content": "RFC 8628 ..." },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 38,
    "completion_tokens": 144,
    "total_tokens": 182
  }
}

Notes

Streaming responses use the same SSE format as OpenAI. The proxy adds two response headers: x-arcana-tier (the license tier that served the request) and x-arcana-cost-credits (credits consumed, when applicable).

GET /v1/models

List the models the proxy can route to. Equivalent to arcana models in the CLI but returns a flat JSON array in OpenAI's shape.

GET /v1/models License or provider key 
curl -H "Authorization: Bearer $ARCANA_LICENSE" \
     https://proxy.arcana.otnelhq.com/v1/models | jq '.data[].id'

GET /v1/balance

Credit balance for the authenticated license subject.

GET /v1/balance License 
{
  "userId": "you@domain.com",
  "credits": 842,
  "dollars": "8.42"
}

credits are integer units (100 credits = $1.00). dollars is a display string.

GET /v1/usage

Daily request counters for the authenticated subject (proxy-side usage meter).

GET /v1/usage License 
{
  "userId": "you@domain.com",
  "date": "2026-07-20"
  /* plus meter fields for the day when present */
}

Free tier also exposes GET /v1/free-usage/sessions/current for the weekly free-session snapshot.

GET /v1/health

Authenticated liveness + identity. Used by the site workspace and by /api/user after device login.

GET /v1/health License / JWT / admin 
{
  "status": "ok",
  "service": "arcana-proxy",
  "user": "you@domain.com",
  "tier": "pro"
}

Admin endpoints

The proxy exposes a small admin surface for license management and provider configuration. All admin endpoints require a bearer from ARCANA_ADMIN_KEYS — a short secret held in the proxy's environment.

Admin keys are not for end users.

These endpoints are intended for the Arcana site and infrastructure tooling. End-user code should call the user-facing endpoints above. The admin secret rotates via Cloudflare Worker secrets — never commit it, never log it.

POST /v1/admin/licenses Admin key

Mint a fresh license. Called by the Arcana site after a user completes device-flow sign-in.

curl -X POST https://proxy.arcana.otnelhq.com/v1/admin/licenses \
  -H "Authorization: Bearer $ARCANA_ADMIN_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "supabaseUserId": "8c2b...",
    "email": "user@domain.com",
    "plan": "pro"
  }'
{
  "licenseKey": "ARCANA-PRO-...",
  "tier": "pro",
  "createdAt": 1721000000000
}
GET /v1/admin/providers Admin key

Read the active provider priority list and which upstreams are configured (no raw API secrets returned).

PUT /v1/admin/providers Admin key

Set provider priority. Body: { "priority": ["openrouter","omniroute","aihubmix","cloudflare"] } (subset / order only).

Also available (auth required, not admin): POST /v1/embeddings, GET /v1/sessions, GET /v1/purchases, GET|PUT /v1/profile, PayPal/billing routes under /v1/pay/*.

Device flow · Pairing

The device-flow login pairs a terminal with a user account without the CLI ever seeing a password. The browser does the sign-in; the CLI waits for the result.

The full sequence, end to end:

  1. Code request. CLI posts to {site}/auth/device/code with { client_id: "arcana-cli" }. The site returns a device_code (UUID, secret) and a user_code (8 chars, public).
  2. Browser open. CLI opens {site}/auth/device?code={user_code} in the user's default browser. The page shows a Supabase sign-in form (email/password, GitHub, Google, or magic link).
  3. Poll loop. CLI starts polling {site}/auth/device/token every interval seconds (server returns interval: 5). While the user is signing in, the response is 400 { error: "authorization_pending" }.
  4. Approve. On successful sign-in, the page calls {site}/auth/device/complete with the user's Supabase ID and email. The site calls the proxy's POST /v1/admin/licenses to mint a fresh license, then writes the resulting license_key back to its KV entry.
  5. License returned. The CLI's next poll returns 200 { access_token: license_key, refresh_token, token_type: "Bearer", expires_in: 31536000 }. The engine also loads /api/user and /api/orgs, then persists the account locally.
Rotate by logging in again

Run arcana console login (and logout if you need a clean slate). Each successful pairing mints a fresh license key via the site → proxy admin path.

user_code format

The user_code is the short, human-typable string the user sees in the browser URL. Format:

  • 8 characters, two groups of 4 separated by a dash: XXXX-XXXX.
  • Alphabet is confusable-free: ABCDEFGHJKLMNPQRSTUVWXYZ23456789. No 0, O, 1, I, or L.
  • ~20 bits of entropy. Combined with the 10-minute TTL and per-IP rate limit on the /code endpoint, brute force is not practical.

Server-side validation regex:

const valid = /^[A-HJ-NP-Z2-9]{4}-[A-HJ-NP-Z2-9]{4}$/.test(userCode)

Error codes

The /auth/device/token poll endpoint returns standard RFC 8628 error codes:

ErrorStatusMeaningCLI behavior
authorization_pending 400 User has not yet approved. Keep polling at interval.
slow_down 400 Polling too fast. Back off interval + 5s. Increase poll interval.
expired_token 400 More than 10 minutes elapsed. Print a hint, exit. User retries with a new arcana console login.
access_denied 400 User explicitly rejected the pairing on the page. Print the rejection, exit.
rate_limited 429 Too many requests from this IP. Retry-After header set. Wait and retry.

The /auth/device/complete endpoint has its own per-user_code rate limit (5 attempts per 5 minutes) to prevent a leaked user_code from being hijacked before the legitimate user finishes signing in.
